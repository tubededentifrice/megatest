# 12 - Diff-Based Route Detection

Status: Draft

## 1. Overview

Route detection is a lightweight static analysis mechanism that identifies new pages/routes in a codebase after a PR merge. It compares detected routes against existing Megatest workflow coverage and surfaces uncovered routes for targeted re-discovery.

Route detection is NOT the same as full AI-powered discovery (spec 08). It is a fast, deterministic, code-scanning step that runs automatically after PR merges. Full discovery is triggered only when the user or system decides to generate workflows for uncovered routes.

---

## 2. Trigger

Route detection is triggered when:

1. A `pull_request.closed` webhook with `merged=true` is received (spec 06, section 4.2).
2. The webhook handler enqueues a route detection job alongside the baseline promotion job.
3. The route detection job runs on a separate lightweight queue (`megatest:route-detection`).

Route detection does NOT consume a run slot. It does not start Docker containers or run browsers. It is pure static analysis on the cloned repository.

### Queue Configuration

```ts
const routeDetectionWorker = new Worker('megatest:route-detection', processRouteDetection, {
  connection: redis,
  concurrency: 4,        // lightweight jobs, higher concurrency is fine
  limiter: {
    max: 1,
    groupKey: 'projectId',
  },
});
```

### Job Data

```ts
interface RouteDetectionJob {
  projectId: string;
  repoUrl: string;
  baseSha: string;           // merge base commit
  mergeSha: string;          // merge commit SHA
  defaultBranch: string;     // e.g. "main"
  installationId: number;    // GitHub App installation ID
  prNumber: number;          // the merged PR number
}
```

## 3. Framework-Specific Parsers

Route detection uses framework-specific parsers to extract route definitions from source code. Each parser scans specific files/directories and outputs a list of route paths.

### 3.1 Next.js (Pages Router)

- **Scan:** `pages/**/*.{js,jsx,ts,tsx}` (excluding `_app`, `_document`, `_error`, `api/`)
- **Rule:** File path maps to route path: `pages/about.tsx` -> `/about`, `pages/blog/[slug].tsx` -> `/blog/:slug`
- **Dynamic segments:** `[param]` -> `:param`, `[...slug]` -> `*slug`
- **Index files:** `pages/blog/index.tsx` -> `/blog`

### 3.2 Next.js (App Router)

- **Scan:** `app/**/page.{js,jsx,ts,tsx}` and `app/**/layout.{js,jsx,ts,tsx}`
- **Rule:** Directory path maps to route: `app/dashboard/page.tsx` -> `/dashboard`
- **Route groups:** `(group)` directories are stripped from the path. Example: `app/(marketing)/about/page.tsx` -> `/about`
- **Dynamic segments:** Same as Pages Router: `[param]` -> `:param`, `[...slug]` -> `*slug`
- **Parallel routes:** `@slot` directories are ignored (they do not create new URL segments).
- **Intercepting routes:** `(.)`, `(..)`, `(...)` prefixes are stripped.

### 3.3 React Router

- **Scan:** Files matching common patterns: `src/router.{js,jsx,ts,tsx}`, `src/routes.{js,jsx,ts,tsx}`, `src/App.{js,jsx,ts,tsx}`
- **Rule:** Parse for `<Route path="..." />`, `createBrowserRouter([...])`, `createRoutesFromElements`
- **Extraction:** Use regex to find `path` prop values:
  ```
  path\s*[:=]\s*["']([^"']+)["']
  ```
  This captures both JSX prop syntax (`path="/dashboard"`) and object property syntax (`path: '/dashboard'`).
- **Nested routes:** When routes are nested, concatenate parent and child paths. Example: parent `path="/settings"` with child `path="profile"` yields `/settings/profile`.
- **Limitation:** Cannot resolve dynamic route configs computed at runtime (e.g., routes loaded from an API or generated by a function). These are silently skipped.

### 3.4 Vue Router

- **Scan:** `src/router/**/*.{js,ts}`, `src/router.{js,ts}`
- **Rule:** Parse for `{ path: '...', component: ... }` in route arrays.
- **Extraction:** Use regex to find `path` values:
  ```
  path\s*:\s*["']([^"']+)["']
  ```
- **Nested routes:** Vue Router supports `children` arrays. Concatenate parent and child paths, same as React Router.

### 3.5 SvelteKit

- **Scan:** `src/routes/**/+page.svelte` and `src/routes/**/+page.{js,ts}`
- **Rule:** Directory path maps to route: `src/routes/about/+page.svelte` -> `/about`
- **Dynamic segments:** `[param]` -> `:param`
- **Route groups:** `(group)` directories are stripped, same as Next.js App Router.
- **Optional parameters:** `[[param]]` -> `:param?`

### 3.6 Remix

- **Scan:** `app/routes/**/*.{js,jsx,ts,tsx}`
- **Rule:** Remix flat routes convention: `app/routes/dashboard.tsx` -> `/dashboard`, `app/routes/blog.$slug.tsx` -> `/blog/:slug`
- **Dot notation:** Dots become path separators except the last one (file extension). Example: `app/routes/settings.profile.tsx` -> `/settings/profile`
- **Dynamic segments:** `$param` -> `:param`, `$.tsx` (splat route) -> `*`
- **Pathless layouts:** Files starting with `_` are layout routes that do not add a URL segment.

### 3.7 Nuxt

- **Scan:** `pages/**/*.vue`
- **Rule:** File path maps to route: `pages/about.vue` -> `/about`
- **Dynamic segments:** `[id].vue` -> `/:id`
- **Index files:** `pages/blog/index.vue` -> `/blog`
- **Catch-all:** `[...slug].vue` -> `/*slug`

### 3.8 Generic Fallback

When no specific framework is detected, or for frameworks without a dedicated parser:

- **Scan:** All `.{js,jsx,ts,tsx,vue,svelte}` files in `src/` and `app/`
- **Rule:** Regex search for URL-like patterns: string literals starting with `/` that look like routes (exclude imports, file paths, API calls).
- **Heuristics for exclusion:**
  - Skip lines containing `import`, `require`, `from`
  - Skip strings matching common file extensions: `.js`, `.css`, `.png`, `.svg`, etc.
  - Skip strings starting with `/api/`, `/static/`, `/assets/`, `/public/`
  - Skip strings containing `://` (full URLs)
- **Confidence:** Low -- many false positives expected. Results are tagged as `framework: 'generic'` with lower priority.

### Framework Detection

Before running parsers, detect the framework:

1. Check `package.json` for framework dependencies: `next`, `react-router-dom`, `vue-router`, `@sveltejs/kit`, `@remix-run/react`, `nuxt`
2. Check for framework-specific config files: `next.config.js`, `next.config.mjs`, `svelte.config.js`, `nuxt.config.ts`, `nuxt.config.js`, `remix.config.js`, `remix.config.mjs`, `vite.config.ts` (with Remix plugin)
3. Run the most specific matching parser(s). Multiple parsers may run if the project uses multiple routing libraries (e.g., Next.js with a separate marketing site using React Router).

### Parser Output

Each parser returns an array of detected routes:

```ts
interface DetectedRoute {
  path: string;              // normalized route path, e.g. "/dashboard"
  sourceFile: string;        // relative path from repo root, e.g. "pages/dashboard.tsx"
  framework: string;         // parser that found it, e.g. "nextjs-pages"
  isDynamic: boolean;        // true if the route contains dynamic segments
  confidence: number;        // 0.0-1.0, lower for generic fallback
}
```

## 4. Diff-Based Optimization

Route detection is optimized to only scan files changed in the merged PR:

1. **Get changed files:**
   ```
   git diff --name-only {base_sha}...{merge_sha}
   ```

2. **Filter to relevant files:** Intersect the changed file list with the active parser's scan patterns. For example, if the project uses Next.js Pages Router, filter to files matching `pages/**/*.{js,jsx,ts,tsx}`.

3. **Fast path:** If no matching files changed, skip route detection entirely. Log "No routing-related files changed" and mark the job as completed immediately.

4. **Classify changes:** For each changed file, determine the change type:
   - `git diff --diff-filter=A --name-only {base_sha}...{merge_sha}` -- new files (added)
   - `git diff --diff-filter=D --name-only {base_sha}...{merge_sha}` -- deleted files
   - `git diff --diff-filter=M --name-only {base_sha}...{merge_sha}` -- modified files

5. **Run the parser on changed files only:**
   - For **new files**: extract routes and check against existing coverage. These are the primary source of uncovered routes.
   - For **modified files**: re-extract routes. If the route path changed (e.g., file renamed or route string updated), treat the new path as potentially uncovered.
   - For **deleted files**: mark previously detected routes from those files as potentially stale by setting `last_seen_sha` but not removing the record.

6. **Full scan fallback:** If a changed file is a router configuration file (e.g., `src/router.ts` for React Router, `src/router/index.ts` for Vue Router), the diff-based optimization is insufficient because a single file change can add or remove many routes. In this case, run a full scan of the configuration file to extract all routes, then diff against previously stored routes.

This optimization ensures route detection completes in seconds even on large codebases.

## 5. Coverage Comparison

After extracting routes from the diff, compare against existing Megatest workflow coverage.

### Extracting Workflow URLs

For each workflow in `.megatest/workflows/*.yml`, extract the URLs from `open:` steps. These represent the routes covered by existing workflows.

Example:

```yaml
# workflows/dashboard.yml
steps:
  - open: http://localhost:3000/dashboard
  - screenshot: dashboard-main
  - open: http://localhost:3000/dashboard/settings
  - screenshot: dashboard-settings
```

Covered routes: `/dashboard`, `/dashboard/settings`

**URL normalization:** Strip the origin (`http://localhost:3000`) and any trailing slash. Convert to lowercase for comparison. Normalize parameter syntax: `${VAR}` in URLs is treated as a wildcard parameter segment.

### Matching Rules

Routes are matched using the following rules, applied in order:

1. **Exact match:** `/about` in workflows covers `/about` in detected routes.

2. **Parameterized match:** Normalize parameter syntax before comparison.
   - Workflow URL `/blog/${SLUG}` matches detected route `/blog/:slug`
   - Detected route `/blog/:slug` matches workflow URL `/blog/my-post` (literal values in workflows match dynamic segments in routes)
   - Parameter names are ignored; only the structure matters.

3. **Prefix match (disabled by default):** `/dashboard` in workflows could cover `/dashboard/settings`. This is disabled by default because it would miss important sub-pages. Can be enabled per-project via a `route_detection.prefix_match` setting.

### Uncovered Routes

Routes present in the codebase but not covered by any workflow are "uncovered". They are candidates for targeted re-discovery.

A route is considered uncovered when:
- No workflow contains an `open:` step whose URL path matches the route.
- The route is not already in the `detected_routes` table with `suggestion_status = 'dismissed'` and the same `source_file` (dismissed routes do not resurface unless the source file changed again in this diff).

## 6. Output

Route detection produces a structured result with metadata about detected and uncovered routes:

```json
{
  "project_id": "proj-uuid",
  "commit_sha": "abc123",
  "pr_number": 42,
  "framework": "nextjs",
  "routes_detected": 14,
  "routes_covered": 11,
  "routes_uncovered": 3,
  "routes_stale": 0,
  "uncovered_routes": [
    {
      "path": "/blog",
      "source_file": "pages/blog/index.tsx",
      "framework": "nextjs-pages",
      "type": "new_file",
      "is_dynamic": false,
      "confidence": 1.0
    },
    {
      "path": "/changelog",
      "source_file": "pages/changelog.tsx",
      "framework": "nextjs-pages",
      "type": "new_file",
      "is_dynamic": false,
      "confidence": 1.0
    },
    {
      "path": "/integrations",
      "source_file": "pages/integrations/index.tsx",
      "framework": "nextjs-pages",
      "type": "new_file",
      "is_dynamic": false,
      "confidence": 1.0
    }
  ],
  "stale_routes": []
}
```

### Storing Results

Uncovered routes are upserted into the `detected_routes` table (spec 03):

| Column | Value |
|--------|-------|
| `route_path` | The detected route path |
| `framework` | Which parser found it |
| `source_file` | The source file where it was detected |
| `first_seen_sha` | Set on first insert only; not updated on subsequent detections |
| `last_seen_sha` | The merge commit SHA (updated on every detection) |
| `covered_by_workflow` | `NULL` (uncovered) |
| `suggestion_status` | `'pending'` |

Routes that were previously detected and are now covered (a workflow was added matching the route) have their `covered_by_workflow` field updated to the matching workflow name.

Routes from deleted files retain their records but are not updated with a new `last_seen_sha`. They age out naturally -- the UI can filter routes that have not been seen since a configurable number of commits or days.

### Upsert Logic

```sql
INSERT INTO detected_routes (project_id, route_path, framework, source_file, first_seen_sha, last_seen_sha, suggestion_status)
VALUES ($1, $2, $3, $4, $5, $5, 'pending')
ON CONFLICT (project_id, route_path) DO UPDATE SET
  framework = EXCLUDED.framework,
  source_file = EXCLUDED.source_file,
  last_seen_sha = EXCLUDED.last_seen_sha,
  updated_at = now()
WHERE detected_routes.suggestion_status != 'dismissed'
   OR detected_routes.source_file != EXCLUDED.source_file;
```

The `WHERE` clause on the upsert ensures:
- Dismissed routes are NOT re-activated unless the source file changed (indicating a meaningful code change that warrants re-evaluation).
- Non-dismissed routes are always updated with the latest SHA and source file info.

## 7. Action on Uncovered Routes

Based on the project's `re_discovery_mode` setting (spec 08, section 8.1):

### Auto-PR Mode (`re_discovery_mode: 'auto_pr'`)

1. Route detection finds uncovered routes.
2. Automatically enqueue a targeted discovery job for those specific routes.
3. The targeted discovery job starts the app and navigates directly to each uncovered route (no full exploration).
4. For each route: take a snapshot, identify the page's purpose, generate a simple workflow (navigate + screenshot).
5. Create a PR with the new workflow files.

**Throttling:** To avoid PR spam, auto-PR mode batches uncovered routes from merges within a configurable window (default: 5 minutes). If multiple PRs merge in quick succession, their uncovered routes are combined into a single discovery job and a single PR.

**PR format:**

```markdown
## Megatest: New Workflow Suggestions

Route detection found {N} new pages not covered by existing workflows.

### New workflows
- `workflows/blog.yml` -- /blog (source: pages/blog/index.tsx)
- `workflows/changelog.yml` -- /changelog (source: pages/changelog.tsx)
- `workflows/integrations.yml` -- /integrations (source: pages/integrations/index.tsx)

### How these were generated
Megatest detected new route files in PR #{pr_number} and automatically
navigated to each page to generate baseline workflows. Review the
generated workflows and merge to enable visual regression testing for
these pages.
```

### Suggest-in-UI Mode (`re_discovery_mode: 'suggest'`, default)

1. Route detection finds uncovered routes.
2. Store them as suggestions in the `detected_routes` table.
3. Show them in the project's Discovery tab (spec 07).
4. User can:
   - Click "Discover" on individual routes to trigger targeted discovery.
   - Click "Discover all" to discover all uncovered routes at once.
   - Click "Dismiss" to hide a suggestion (`suggestion_status` set to `'dismissed'`).
   - Dismissed routes do not resurface unless the source file changes again in a subsequent merge.

## 8. Targeted Discovery

Targeted discovery is a lightweight variant of full discovery (spec 08) that focuses on specific routes rather than exploring the entire application.

### Flow

1. **Setup:** Reuse the same Docker setup as full discovery (clone, install, serve). The container setup is identical -- it uses the project's existing `config.yml` to install dependencies and start the dev server.

2. **Navigate directly:** Instead of exploring the entire app, navigate directly to each target route. No exploration phase, no link following.

3. **Per-route process:**

   a. `open` the route URL (replace the route path into the `serve.ready` base URL).

   b. `snapshot -i` to understand the page structure.

   c. Identify key visual sections from the snapshot.

   d. Check for authentication gates: if the page redirects to a login URL or shows auth-required content, and a `login` include exists in `.megatest/includes/`, prepend `include: login` to the workflow.

   e. Generate a simple workflow:
      ```yaml
      name: blog
      description: Blog listing page
      steps:
        - open: http://localhost:3000/blog
        - wait: 1000
        - screenshot: blog-page
      ```

   f. For pages with distinct sections (detected via snapshot analysis), generate multiple screenshots:
      ```yaml
      name: integrations
      description: Integrations marketplace
      steps:
        - open: http://localhost:3000/integrations
        - wait: 1000
        - screenshot: integrations-hero
        - scroll: down 500
        - screenshot: integrations-list
      ```

4. **Generate workflow YAML files:** One file per discovered route. File names are derived from the route path: `/blog` -> `blog.yml`, `/settings/profile` -> `settings-profile.yml`.

5. **Apply based on storage mode:** The generated files are applied through the same `POST /api/v1/discoveries/:id/apply` endpoint used by full discovery. Behavior depends on the project's `config_storage_mode`:
   - **repo:** PR is created on the project repository.
   - **server:** Config files are stored in the database.
   - **config_repo:** PR is created on the config repository.

### Performance

Targeted discovery is significantly faster than full discovery:

| Metric | Full Discovery | Targeted Discovery |
|--------|---------------|--------------------|
| Typical duration | 2-5 minutes | 15-60 seconds |
| Pages visited | 10-50 | 1-10 (only target routes) |
| LLM calls | 20-100 | 1-5 per route |
| Docker container time | 3-6 minutes | 1-2 minutes |

The speed improvement comes from skipping the exploration phase entirely. Targeted discovery knows exactly which URLs to visit.

### Dynamic Route Handling

For routes with dynamic segments (e.g., `/blog/:slug`), targeted discovery must synthesize a test URL:

1. Use a placeholder value: `/blog/:slug` -> `/blog/test-post`
2. Navigate to the placeholder URL.
3. If the page returns a 404 or error, note this in the generated workflow as a comment:
   ```yaml
   name: blog-detail
   description: Blog post detail page (dynamic route, may need a valid slug)
   # NOTE: This route uses a dynamic segment. The placeholder URL may not
   # resolve to a real page. Update the open URL with a valid slug.
   steps:
     - open: http://localhost:3000/blog/test-post
     - wait: 1000
     - screenshot: blog-detail-page
   ```
4. If the page renders successfully with the placeholder, use it as-is.

## 9. Execution Details

### Cloning for Route Detection

Route detection requires access to the repository source code but does NOT need to build or run the application. The clone is lightweight:

```bash
git clone --depth=1 --branch={defaultBranch} \
  https://x-access-token:{token}@github.com/{owner}/{repo}.git \
  /tmp/megatest-rd-{jobId}/repo

cd /tmp/megatest-rd-{jobId}/repo
git fetch --depth=1 origin {mergeSha}
git checkout {mergeSha}
```

The clone is performed in a temporary directory (`/tmp/megatest-rd-{jobId}/`) that is cleaned up after the job completes.

### Processing Steps

The `processRouteDetection` function follows this sequence:

1. **Clone** the repository at the merge commit SHA.
2. **Detect framework** by inspecting `package.json` and config files.
3. **Get changed files** via `git diff --name-only {baseSha}...{mergeSha}`.
4. **Filter** changed files to routing-relevant paths.
5. **Fast path**: if no relevant files changed, complete immediately.
6. **Run parser(s)** on changed files (or full router config files if applicable).
7. **Read existing workflows** from `.megatest/workflows/*.yml` (or from config source per `config_storage_mode`).
8. **Compare** detected routes against workflow coverage.
9. **Upsert** uncovered routes into `detected_routes` table.
10. **Act** on uncovered routes per `re_discovery_mode` (enqueue targeted discovery or store suggestions).
11. **Cleanup** temporary directory.

### Error Handling

| Failure | Behavior |
|---------|----------|
| Clone failure | Job fails. Retry once (BullMQ `attempts: 2`). |
| No `package.json` found | Skip framework detection, use generic fallback parser. |
| Parser error (invalid syntax in route file) | Log warning, skip the problematic file, continue with other files. |
| No `.megatest/workflows/` directory | All detected routes are marked uncovered (no coverage exists). |
| Database write failure | Job fails. Retry once. |

Parser errors are non-fatal. A single malformed route file should not prevent detection of routes in other files. Errors are logged with the file path and error details for debugging.

### Timeouts

| Scope | Limit |
|-------|-------|
| Total job | 60 seconds |
| Clone | 30 seconds |
| Parser execution (per file) | 5 seconds |

Route detection jobs that exceed 60 seconds are terminated. This should rarely happen since the work is pure file I/O and regex matching with no network calls beyond the initial clone.

## 10. Limitations

- **Dynamic routes:** Routes like `/blog/:slug` are detected but targeted discovery can only test with a placeholder URL (e.g., `/blog/test-post`). The generated workflow may need manual adjustment to use a valid slug.

- **API routes:** API-only routes (no UI) are excluded from detection. Filters: exclude paths starting with `/api/`, Next.js `pages/api/`, Remix resource routes (files exporting only `loader`/`action` without a default component export).

- **Authenticated routes:** Routes behind authentication are detected but targeted discovery may fail if no login credentials are configured. The suggestion UI notes this: "This route may require authentication. Ensure test credentials are configured before discovering."

- **Framework coverage:** Only frameworks with dedicated parsers (sections 3.1-3.7) produce high-confidence results. The generic fallback (section 3.8) has high false-positive rates and its results are tagged accordingly.

- **Computed routes:** Routes generated dynamically at runtime (e.g., from a database query, feature flags, or programmatic route generation) cannot be detected by static analysis. These routes are invisible to the parser.

- **Monorepos:** For monorepos with multiple apps, route detection should be scoped to the app directory. The project config's `app_root` field (if set) restricts file scanning to that subdirectory. If `app_root` is not set, detection scans from the repository root, which may produce false positives from sibling apps.

- **Route aliases and redirects:** Server-side redirects and route aliases defined outside the routing framework (e.g., in `next.config.js` `rewrites`/`redirects`, nginx config, or middleware) are not detected. Only framework-native route definitions are parsed.

- **Internationalized routes:** i18n routing (e.g., Next.js `i18n` config, Nuxt `i18n` module) may produce multiple route variants per page (`/about`, `/fr/about`, `/de/about`). The parser detects the base route but does not expand locale variants. Locale-specific routes must be manually added if coverage is desired.
